# Simulator — Mathematical & Code Inference

This document covers the simulator layer that sits on top of the MDP environment. The environment defines the **rules of the game**; the simulator is what actually **plays the game for $T$ steps, records what happened, and evaluates the outcome**.

I won't re-derive MDP tuples, state/action spaces, Markov chains, PRNG seeding, or $L_1$/$L_2$ penalties here — those are already handled in `dynamics_formulation.md` and `mdp_environment.md`. This file focuses on what's **new** at the simulator layer: **policies**, **finite horizons**, **cumulative return**, and the financial decomposition of **Baseline vs. Alpha PnL**.

---

# Part I — Mathematical and Statistical Inference

## 1. Finite-Horizon MDPs

The environment I built is episodic — it doesn't run forever. Every simulation terminates after exactly $T$ steps, where $T$ is the `horizon`.

This makes it a **finite-horizon MDP**, defined by the tuple:

$$(S, A, P, R, T)$$

The presence of $T$ is what lets me skip the discount factor $\gamma$ I talked about in `dynamics_formulation.md`. The infinite sum never happens, so convergence is not a concern — the return is a finite sum by construction.

> **Why finite horizon for trading?** Energy markets settle on daily, hourly, or sub-hourly cycles. A "day" or a "delivery window" is a natural episode. I don't need to reason about rewards a thousand years from now.

---

## 2. Policies ($\pi$)

A **policy** is the agent's brain. It's the mapping function that dictates *what action to take in a given state*. The simulator supports two flavors.

### 2.1 Deterministic Policy

A direct, absolute mapping from a state to a single action:

$$a_t = \pi(s_t)$$

Same state in → same action out, every time. In my code this is literally a Python dictionary: `policy[s]`. The lookup is the policy.

### 2.2 Stochastic Policy (Uniform Baseline)

A probability distribution over actions, conditional on state:

$$P(A_t = a \mid S_t = s)$$

For my random baseline, this distribution is **uniform** over the action space — every action gets equal probability:

$$P(A_t = a \mid S_t = s) = \frac{1}{|A|} = \frac{1}{3}$$

The uniform policy is deliberately dumb. Its job is to be a **null hypothesis** — if my "optimal" policy can't beat random action selection, the optimal policy isn't optimal.

---

## 3. Cumulative Return — The Objective Function

In RL the agent isn't evaluated on any single reward; it's evaluated on the **sum of rewards** across the episode. Since I'm running finite-horizon without a discount factor, I compute the **undiscounted return**:

$$G = \sum_{t=0}^{T-1} R_{t+1}$$

This $G$ is what my code stores as `total_reward` and `alpha_pnl`. It's a single scalar that answers: *"Across this entire episode, how much did the agent earn?"*

> **Undiscounted is a modeling choice, not laziness.** I'm inside a short-enough horizon that $\gamma \to 1$ anyway. If I stretched the horizon across multiple trading days, I'd reintroduce $\gamma$ to capture time-value-of-money.

---

## 4. Financial Mathematics — Baseline vs. Alpha

This is the most important idea in the simulator, and it's borrowed directly from quantitative finance. The simulator isolates **the specific value generated by the battery** from the value generated by the underlying site.

### 4.1 Baseline PnL (Net Load)

The **Baseline** is the counterfactual — what would the site earn or pay if the battery didn't exist? The site still generates power, still consumes `DEMAND`, and still transacts the difference at the market price $P_t$:

$$\text{Baseline}_t = P_t \times (\text{Generation}_t - \text{Demand})$$

If generation exceeds demand, the site sells the surplus. If demand exceeds generation, the site buys the shortfall. No decision-making involved — just passive net-load arithmetic.

### 4.2 Alpha ($\alpha$) — The Battery's Contribution

In finance, **Alpha** is the excess return of a strategy relative to a passive benchmark. In my system, Alpha is the pure economic value generated by the battery's **arbitrage** — charging when power is cheap, discharging when expensive.

$$\alpha_t = R_t \quad \text{(the reward returned by the MDP environment)}$$

The environment's reward *already is* the alpha, because the environment only scores what the battery does — not the underlying site's net load.

### 4.3 Total PnL — The Decomposition

The full picture is a sum of the two streams:

$$\text{Total PnL} = \sum_{t=0}^{T-1} \left( \text{Baseline}_t + \alpha_t \right)$$

This decomposition is what makes the simulator diagnostically useful. If Total PnL is bad, I can immediately ask: *was the site just unlucky (baseline dragged us down), or did the policy actively destroy value (negative alpha)?* Without the split, I can't tell.

## 2. Simulator Initialization

```python
class Simulator:
    def __init__(self, env: MDPEnvironment, policy: dict, horizon: int) -> None:
        self.env     = env
        self.policy  = policy
        self.horizon = horizon
```

- **`self.env`** — holds the transition dynamics $P$ and reward function $R$.
- **`self.policy`** — the **deterministic policy** $\pi$ from §2.1, stored as a dict where keys are states and values are optimal actions.
- **`self.horizon`** — explicitly sets $T$. The simulator *is* a finite-horizon construct; without this, no episode.

## 4. The Policy Execution Methods

These are the **public methods** I actually call. Both delegate to `_run`, injecting different `action_fn` lambdas to change the agent's behavior.

```python
    def run_policy(self) -> SimulationSummary:
        return self._run(action_fn=lambda s: self.policy[s])
```

**Deterministic execution.** The lambda `lambda s: self.policy[s]` is literally the mathematical mapping $a = \pi(s)$ — a dictionary lookup. No randomness, no exploration, no deliberation. Same state always produces the same action.

```python
    def run_random_policy(self, seed: int = 0) -> SimulationSummary:
        rng = np.random.default_rng(seed)
        return self._run(action_fn=lambda s: int(rng.choice(ACTIONS)))
```

**Stochastic execution (uniform baseline).** The lambda `lambda s: int(rng.choice(ACTIONS))` implements the uniform stochastic policy from §2.2. Notice that `s` is **ignored** inside the lambda — the random policy doesn't even look at the state, which is the whole point. It's the dumbest possible agent, and that's the value it brings as a benchmark.

The `seed` argument ensures reproducibility: seed `0` will always produce the same sequence of "random" actions, so two comparisons against the random baseline are actually comparing against the *same* baseline.

---

## Closing Note

The simulator is intentionally thin. All the physics lives in the environment, all the randomness of the market lives in the dynamics module, and all the *policy behavior* lives in whatever `action_fn` I pass in. The simulator's only job is to **drive the clock forward, record the journey, and split the PnL into its causal components**. That separation is what lets me swap policies without rewriting anything else — exactly what I want before I start plugging in Q-learning or Proximal Policy Optimizatio (PPO).
